# Patient Microservice

Continuously Delivered Microservice for managing patient records, built with Spring Boot and MongoDB.

---

## Tech Stack

| Technology | Purpose |
|--------|---|
| Java 21 | Programming language |
| Spring Boot 4.0.6 | Application framework |
| Spring Data MongoDB | Database access |
| Spring Validation | Request validation |
| Lombok | Boilerplate reduction |
| JUnit 5 | Unit and integration testing |
| Mockito | Mocking in unit tests |
| Docker | Containerization |
| GitLab CI | Continuous integration pipeline |

---

## Domain Model

A `Patient` has the following properties:

| Field | Type | Validation                           |
|---|---|--------------------------------------|
| `id` | String | Auto-generated by MongoDB            |
| `firstName` | String | Required                             |
| `lastName` | String | Required                             |
| `dateOfBirth` | LocalDate | Required                             |
| `contactNumber` | String | Required: 7-15 digits                |
| `emailAddress` | String | Required: valid email format, unique |
| `gender` | Enum | Required: `MALE`, `FEMALE`, `OTHER`  |

## REST API Endpoints

Base URL: `http://localhost:8080/api/patients`

| Method | Endpoint | Description | Success Response |
|---|---|---|---|
| GET | `/api/patients` | Get all patients | 200 OK |
| GET | `/api/patients/{id}` | Get a patient by ID | 200 OK |
| POST | `/api/patients` | Add a new patient | 201 Created |
| PUT | `/api/patients/{id}` | Update a patient | 200 OK |
| DELETE | `/api/patients/{id}` | Delete a patient | 204 No Content |

### Error Responses

| Status | Reason                                               |
|---|------------------------------------------------------|
| 400 Bad Request | Validation failed - missing or invalid fields        |
| 404 Not Found | Patient with the given ID does not exist             |
| 409 Conflict | A patient with the same email address already exists |

### Example Request — Add Patient

```bash
curl -X POST http://localhost:8080/api/patients \
  -H "Content-Type: application/json" \
  -d '{
    "firstName": "Alice",
    "lastName": "Müller",
    "dateOfBirth": "1985-03-15",
    "contactNumber": "+4366412345678",
    "emailAddress": "alice.mueller@gmail.com",
    "gender": "FEMALE"
  }'
```

### Example Response

```json
{
  "id": "64b1f2c3e4b0a1d2e3f4a5b6",

  "dateOfBirth": "1985-03-15",
```

## Running Locally

### Prerequisites

- Java 21+
- Maven 3.8.7+
- Docker

### Steps

**1. Start MongoDB:**

```bash
docker run -d --name patient-mongo -p 27017:27017 -v mongo_data:/data/db mongo:7.0
```

**2. Build the project:**

```bash
mvn clean package -DskipTests
```

**3. Run the application:**

Run the `main()` method in `PatientmicroserviceApplication.java` from your IDE, or:

```bash
mvn spring-boot:run -Dspring-boot.run.mainClass=org.example.patientmicroservice.PatientMicroserviceApplication
```

The service starts on `http://localhost:8080`.

**4. Run the demo client:**

Run the `main()` method in `PatientServiceClient.java` from your IDE, or:

```bash
mvn exec:java -Dexec.mainClass="org.example.patientmicroservice.client.PatientServiceClient"
```

## Running with Docker Compose

Start the app, MongoDB, and tests all together:

```bash
docker compose up -d --build
```

Run the application:

Run the `main()` method in `PatientmicroserviceApplication.java` from your IDE, or:

```bash
mvn spring-boot:run -Dspring-boot.run.mainClass=org.example.patientmicroservice.PatientMicroserviceApplication
```

Run only the tests:

```bash
docker compose run --rm test
```

Stop and clean up:

```bash
docker compose down -v
```

**Unit tests only**:

**Integration tests only**:

## CI Pipeline

The GitLab CI pipeline defined in `.gitlab-ci.yml` has two stages:

| Stage | Description |
|---|---|
| `build` | Compiles the project, builds the Docker image, saves it as a pipeline artifact |
| `test` | Loads the image, starts MongoDB and the app, runs all tests |

The pipeline uses Docker in Docker (DIND) and passes the built image between stages using a `.tar` artifact to avoid rebuilding in the test stage.
